# `/teach` Command — First-Time Design Context Setup

## Context

Impeccable has `/impeccable teach` as a mandatory first-time setup that scans the project, asks UX questions, and writes a `.impeccable.md` design context file. All subsequent commands reference this context for project-specific guidance.

klamben currently has no equivalent. All 21 commands apply generic Flutter rules without knowing the project's brand, target audience, platform focus, or existing theme setup. A `/teach` command would make every other command significantly more useful by grounding advice in the project's actual design decisions.

## What `/teach` does

Three-step flow (mirrors impeccable's approach, adapted for Flutter):

### Step 1: Codebase Scan (automatic, no user input)

Scan the Flutter project to discover existing design patterns:

- **Theme setup:** Does `ThemeData` exist? What's the seed color? Is dark mode configured? Which font is used?
- **Spacing:** Are spacing tokens defined (ThemeExtension, const file)? What grid values are used?
- **Platform:** Does the project import `cupertino_icons`? Are there `Platform.isIOS` checks? Any `.adaptive` constructors?
- **Dependencies:** `google_fonts`, `flutter_svg`, `cached_network_image`, `flutter_localizations`, `flutter_platform_widgets`?
- **Localization:** Is `l10n.yaml` present? Are ARB files set up?
- **Accessibility:** Any `Semantics` widgets? `tooltip` on `IconButton`s?
- **Package info:** app name from `pubspec.yaml`, min SDK version

Output: internal summary of what was found. Skip questions whose answers are already evident.

### Step 2: Ask UX Questions (interactive, one at a time)

Only ask questions the codebase scan couldn't answer. Prefer multiple choice.

**Questions (skip if scan answers them):**

1. **Target platforms** — "Which platforms does this app target?"
   - Android only
   - iOS only
   - Both Android + iOS (platform-adaptive)
   - Both but Material-only (skip Cupertino)

2. **Brand personality** — "Pick 2-3 words that describe this app's personality:"
   - Professional / Playful / Minimal / Bold / Warm / Technical / Premium / Friendly
   - (or open-ended)

3. **Primary audience** — "Who uses this app?"
   - General consumers
   - Business/enterprise users
   - Developers/technical users
   - Kids/education
   - (or open-ended)

4. **Accessibility tier** — "What level of accessibility do you target?"
   - WCAG AA (recommended)
   - WCAG AAA (strict)
   - Best-effort (no formal target)

5. **Design maturity** — "What's the state of this project's design?"
   - Greenfield — no design system yet
   - Has brand colors + fonts but no formal tokens
   - Has a design system (Figma/tokens/components)
   - Following an external design spec

6. **Seed color** — "What's your primary brand color?" (if no `ColorScheme.fromSeed` found in scan)
   - Hex input or skip for default

### Step 3: Write `.klamben.md`

Write a structured design context file to project root:

```markdown
# klamben Design Context

> Generated by `/teach`. Edit freely — klamben commands reference this file.

## Project

- **App:** MyApp
- **Platforms:** Android + iOS (platform-adaptive)
- **Min SDK:** Flutter 3.22 / Dart 3.4

## Brand

- **Personality:** Professional, Minimal
- **Audience:** Business/enterprise users
- **Seed color:** #6750A4

## Design Decisions

- **Theme:** ColorScheme.fromSeed with dark mode ✓
- **Font:** Inter via google_fonts ✓
- **Spacing:** 4/8pt grid via ThemeExtension ✓ (or ✗ — not yet set up)
- **Platform adaptation:** Material + Cupertino adaptive ✓ (or ✗)
- **Localization:** flutter_localizations + ARB ✓ (or ✗)
- **Accessibility target:** WCAG AA

## Existing Patterns Found

- ThemeData at `lib/theme/app_theme.dart`
- Spacing tokens at `lib/theme/spacing.dart`
- No Semantics widgets found — `/check-a11y` recommended

## Preferences

- [Any custom preferences the user mentioned during teach]
```

### How other commands use `.klamben.md`

Every command's prompt should include: "Before starting, check if `.klamben.md` exists in the project root. If it does, read it and apply the design context to your work."

Examples:
- `/colorize` checks seed color from `.klamben.md` instead of asking
- `/typeset` checks font preference
- `/audit` adjusts severity based on accessibility tier (WCAG AAA makes contrast rules stricter)
- `/adapt` checks platform targets (skip Cupertino if Android-only)
- `/critique` uses brand personality to evaluate if UI matches tone

## Out of scope

- `.klamben.md` parser in the CLI detector (CLI uses rules.json, not design context)
- Interactive `/teach` flow in the CLI (it's an AI harness command, not a CLI feature)
- `/re-teach` or `/forget` commands (user can just edit/delete `.klamben.md`)
